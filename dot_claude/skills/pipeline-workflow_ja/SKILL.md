# パイプラインワークフロー

Inspectorによる品質ゲートを各ステージ間に配置した、順次実行型マルチエージェントパイプライン。各エージェントの役割は1つだけ。利用可能な場合はAnthropic公式エージェントを使用する。

## パイプラインフロー

```
Interviewer → [Inspector] → Explorer → Planner → [Inspector] → Implementer → [Inspector] → Simplifier → Review（4エージェント並列）→ [Inspector] → Tester → [Inspector] → 完了
```

## エージェント一覧

| ステージ | エージェント | ソース | 役割 |
|---------|-----------|--------|------|
| 要件ヒアリング | `pipeline--interviewer_ja` | カスタム | ユーザーインタビュー、仕様書作成 |
| コードベース探索 | `feature-dev--code-explorer_ja` | 公式 | コードベース深掘り調査 |
| 設計・計画 | `feature-dev--code-architect_ja` | 公式 | アーキテクチャ設計、実装計画 |
| 実装 | `pipeline--implementer_ja` | カスタム | TDDによるコーディング |
| コード簡素化 | `pr-review-toolkit--code-simplifier_ja` | 公式 | コードの明確性を磨く |
| レビュー（汎用） | `pr-review-toolkit--code-reviewer_ja` | 公式 | コード品質、CLAUDE.md準拠 |
| レビュー（エラー） | `pr-review-toolkit--silent-failure-hunter_ja` | 公式 | エラーハンドリング監査 |
| レビュー（コメント） | `pr-review-toolkit--comment-analyzer_ja` | 公式 | コメント正確性 |
| レビュー（型） | `pr-review-toolkit--type-design-analyzer_ja` | 公式 | 型設計品質（条件付き） |
| テスト | `pipeline--tester_ja` | カスタム | テスト設計・実行・カバレッジ |
| 品質ゲート | `pipeline--inspector_ja` | カスタム | 成果物検査 |

## 実行ステップ

### ステップ 0: セットアップ

隔離されたワークスペースを作成する:
```
EnterWorktreeを使用するか、隔離用の新しいgitブランチを作成する。
```

### ステップ 1: 要件ヒアリング

`pipeline--interviewer_ja` でエージェントを起動:
- 提供する情報: ユーザーからの機能説明
- InterviewerはAskUserQuestionでユーザーに質問する
- Interviewerが要件仕様書を作成するまで待つ

### ステップ 1.5: 仕様書検査

`pipeline--inspector_ja` でエージェントを起動:
- 提供する情報: 要件仕様書テキスト + 指示「検査: Interviewer後」
- FAIL: 不備リストを新しいInterviewerエージェントに送り修正させる
- PASS: 仕様書をユーザーに提示し最終確認を得る
- 最大3回のリトライ後、ユーザーにエスカレーション

### ステップ 2: コードベース探索

`feature-dev--code-explorer_ja` で2〜3個のエージェントを**並列**起動し、それぞれ異なるフォーカスを持たせる:
- エージェント1: 要件に類似した機能を見つけ、実装をトレース
- エージェント2: 関連領域のアーキテクチャと抽象化をマッピング
- エージェント3: テストパターン、拡張ポイント、既存ユーティリティを特定（任意）

各エージェントに提供: コンテキストとして要件仕様書テキスト。

調査結果を収集し、コードベースコンテキストサマリーを作成する。このサマリーをPlannerに提供する。

### ステップ 3: 実装計画

`feature-dev--code-architect_ja` でエージェントを起動:
- 提供する情報: 要件仕様書 + コードベース探索サマリー
- 以下の形式で出力するよう追加指示する:
  - タスクチェックリスト: 各タスクに対象ファイル、テストファイル、検証コマンド、完了条件
  - 各タスク内でTDD順序（テスト→実装→検証）
  - 仕様書のすべての要件が少なくとも1つのタスクに対応すること
- アーキテクトが実装計画/ブループリントを作成するまで待つ

### ステップ 3.5: 計画検査

`pipeline--inspector_ja` でエージェントを起動:
- 提供する情報: 実装計画テキスト + 要件仕様書 + 指示「検査: Planner後」
- FAIL: 不備リストを新しいアーキテクトエージェントに送り修正させる
- PASS: 計画をユーザーに提示し承認を得る
- 最大3回のリトライ後、ユーザーにエスカレーション

### ステップ 4: 実装

`pipeline--implementer_ja` でエージェントを起動:
- 提供する情報: 実装計画の全文（テキストとして渡す、ファイル参照させない）
- ImplementerはTDDでタスクを順に実行する
- Implementerが完了レポートを作成するまで待つ

### ステップ 4.5: 実装検査

`pipeline--inspector_ja` でエージェントを起動:
- 提供する情報: Implementerのレポート + 計画 + 指示「検査: Implementer後」
- プロジェクトのリンター、型チェッカー、依存関係チェッカーがあれば実行する
- FAIL: 不備リストを新しいImplementerエージェントに送る
- PASS: 簡素化に進む
- 最大3回のリトライ後、ユーザーにエスカレーション

### ステップ 5: コード簡素化

`pr-review-toolkit--code-simplifier_ja` でエージェントを起動:
- 提供する情報: Implementerのレポートからの変更ファイル一覧
- コードの明確性を磨きつつ機能を保持する
- このステージはポリッシュ — 動作を変更してはいけない

### ステップ 6: コードレビュー（4エージェント並列）

以下のエージェントを1つのメッセージで**並列**起動する:

**常に実行:**
1. `pr-review-toolkit--code-reviewer_ja` — 汎用コード品質、CLAUDE.md準拠、バグ検出
2. `pr-review-toolkit--silent-failure-hunter_ja` — エラーハンドリング監査、サイレントフェイル検出

**条件付きで実行:**
3. `pr-review-toolkit--comment-analyzer_ja` — コメント/docstringが追加・修正された場合
4. `pr-review-toolkit--type-design-analyzer_ja` — 新しい型/クラスが導入された場合

各エージェントに提供: git diff、実装計画、CLAUDE.mdの内容。

**結果の集約**: 全レポートを収集し、統一レビューにマージ:
- いずれかのエージェントがCritical問題を報告: 全体FAIL → 問題を新しいImplementerエージェントに送り、レビューを再実行
- 全エージェントがCritical問題なし: 全体PASS → 検査に進む
- 最大3回のレビュー-修正ループ後、ユーザーにエスカレーション

### ステップ 6.5: レビュー検査

`pipeline--inspector_ja` でエージェントを起動:
- 提供する情報: 全レビューレポート（マージ済み）+ 指示「検査: Reviewer後」
- FAIL: 不備を該当するレビューエージェントに戻す
- PASS: テストに進む
- 最大3回のリトライ後、ユーザーにエスカレーション

### ステップ 7: テスト

`pipeline--tester_ja` でエージェントを起動:
- 提供する情報: 変更ファイル一覧、実装計画のテスト要件、Implementerのテストレポート
- FAILの場合: 失敗を新しいImplementerエージェントに送り、その後Testerを再実行
- PASSの場合: 検査に進む

### ステップ 7.5: テスト検査

`pipeline--inspector_ja` でエージェントを起動:
- 提供する情報: Testerのレポート + 要件仕様書（受け入れ基準用）+ 指示「検査: Tester後」
- FAIL: 不備を新しいTesterエージェントに戻す
- PASS: 完了処理に進む
- 最大3回のリトライ後、ユーザーにエスカレーション

### ステップ 8: 完了

`finishing-a-development-branch` スキルを使用して:
- 適宜PRを作成またはマージ
- 使用した場合はworktreeをクリーンアップ

## ハンドオフルール

1. オーケストレーター（あなた）がエージェント間のコンテキストをテキストとして中継する — エージェントは互いのファイルを読まない
2. 各エージェントに常に仕様書/計画の全文を提供する — 省略しない
3. 各エージェントの出力が次のステージの入力になる
4. Inspectorには常に: 検査対象の成果物 + 関連する参照文書 + 検査モードを提供する
5. Explorerの調査結果はサマリーにまとめてPlannerに提供する
6. レビューレポートはInspector用に統一レポートにマージする

## フィードバックループルール

- 各ステージ最大3回のリトライ後、AskUserQuestionでユーザーにエスカレーション
- 各リトライで新しいエージェントインスタンスを作成する（再利用しない）
- Inspectorの不備リストをリトライの主要入力として提供する
- すべてのリトライに元のコンテキスト（計画/仕様書）を含める

## 絶対にやってはいけないこと

- Interviewerステージを省略しない
- 要件と計画の後のユーザー確認を省略しない
- レビューエージェントにコードを修正させない
- Testerにプロダクションコードを修正させない
- Inspectorに何も修正させない
- FAIL検査を修正せずに次に進まない
- 3回のリトライループを超えてユーザーにエスカレーションしない
- Simplifierを省略しない — レビューの指摘を減らす効果がある
